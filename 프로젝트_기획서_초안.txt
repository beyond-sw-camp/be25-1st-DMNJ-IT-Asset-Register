IT-Asset Ledger 프로젝트 기획서(초안)

1) 서비스 개요
- 사내 IT 자산(노트북/모니터/태블릿/모바일 단말 등)의 조달 → 입고/검수 → 자산화 → 대여/반납 → 반납 검수 → 보증금/연체/파손 정산 전 과정을 단일 데이터 모델로 관리한다.
- “누가 어떤 자산을 언제부터 사용하는지”, “반납 검수 결과가 무엇인지”, “비용 정산을 어떤 근거로 산정했는지”를 증빙 가능한 형태(검수 증빙·정산 라인·원장)로 추적 가능하게 한다.
- 멀티테넌시(company_id)와 역할 기반 흐름 분리로 운영 리스크(분실/중복 지급/재고 불일치/정산 분쟁)를 낮춘다.

2) 타겟 사용자(역할)
- 고객사 구매담당: 발주/승인/계약/공급사 대응
- 공급사 담당자: 수주/출고/납품/불량(RMA)
- 고객사 자산관리자: 입고검수/자산등록/재고/배정
- 렌탈/임차사 담당자: 대여·반납 요청/확인
- 정산·회계 담당자: 보증금 환급/차감, 연체료/수리비 정산 확정

3) 문제 정의(운영 병목)
- 대여·반납 기록이 메신저/엑셀/개인 기록으로 분산되어 자산 현황과 이력의 신뢰도가 떨어진다.
- 반납 검수 기준 편차로 파손 책임과 수리비 산정에서 분쟁이 생긴다.
- 보증금/연체료 산정이 계약·검수 이력과 분리되면 정산 근거(감사/Audit) 재현이 어렵다.

4) 프로젝트 시나리오(요약)
- 구매담당: PO 생성/승인/전송 → 공급사 수락 → 출고/배송완료 → 입고검수 → 불량은 RMA, 합격은 자산화로 이동
- 자산관리자: 입고 등록 → 검수 체크리스트로 합격/불량 기록 → 합격 건에 시리얼/자산태그 부여 후 자산 등록 → AVAILABLE 전환
- 임차사: 대여 요청 → 계약(기간/보증금/정책) 확정 → 자산 할당/출고 → 반납 요청/접수 → 반납 검수
- 정산담당: 검수 결과 기반으로 연체/파손 판단 → 정책 버전 적용 → 정산 라인 생성 → 보증금 원장 환급/차감 → 정산 확정

5) 필요 기능
- 회원/권한: 회사 소속, 역할 부여, 권한 기반 접근제어
- 조달/공급사: 공급사/PO/품목/배송/납품 상태, RMA
- 입고/검수/자산화: 입고 등록, 검수(항목/판정/증빙), 자산 등록(시리얼/태그/상태)
- 대여/반납: 계약/보증금, 자산 할당, 출고/인수확인, 반납 요청/접수/검수
- 정산: 정책(버전/유효기간), 정산서/정산라인, 보증금 원장(입금/차감/환급)
- 조회/리포트: 재고/사용중/미반납/연체/파손/정산 현황

6) 데이터 설계(초기 엔터티 방향)
- 테넌시/권한: company, user, role, user_role
- 조달: vendor, purchase_order, po_item, shipment, receiving, rma
- 자산: asset_model, asset, asset_status_history
- 대여/반납: rental_contract, rental_item, handover, return_request, return_receipt
- 검수 증빙: inspection, inspection_item, evidence_file
- 정산: fee_policy, fee_rule, settlement, settlement_line, deposit_account, deposit_ledger

7) 설계 원칙(감점 방지)
- 자산 유니크: serial_no, asset_tag(회사 단위 유니크)
- 상태전이: 현재 상태 + 이력 테이블로 추적(상태 역전 방지)
- 정산 재현성: 정산 라인에 정책 버전/기준값/산출 결과를 함께 저장(정책 변경 시 과거 정산 불변)
- 검수-정산 연결: 검수 결과(항목/판정/증빙) ↔ 정산라인(차감사유) 관계를 명시

8) 서비스 플로우(텍스트)
- 조달(PO) → 입고/검수 → 자산화 → 대여(계약/보증금/할당/출고) → 반납(요청/접수/검수) → 정산(라인/원장/확정) → 자산 가용 복귀

9) 다이어그램(작성 권장)
- Use-Case Diagram: 역할별 유스케이스 묶음(구매/공급사/자산/임차/정산)
- Sequence Diagram(2개): 대여 시퀀스, 반납~정산 시퀀스
